# Backend Performance Budget

## 목표

상세페이지는 이미지와 인터랙션이 많은 화면이므로 백엔드는 가볍고 예측 가능해야 합니다.

핵심 원칙:

- 필요한 데이터만 내려준다.
- 큰 HTML 본문과 예약 정보 응답을 분리한다.
- 예약 가능일 조회 범위를 제한한다.
- 금액/좌석 계산은 서버에서 하되, 쿼리는 읽기 쉽게 유지한다.
- 기존 DB를 무리하게 새 구조로 바꾸지 않는다.

## 상품 상세 API 모드

`GET /api/products/:id`는 `mode` query를 받는 구조를 권장합니다.

| Mode | 용도 | 포함 데이터 |
| --- | --- | --- |
| `reservation` | 상세페이지 예약 모듈 초기화 | 상품명, 기존 pid, 옵션, 여권/룸정보 필요 여부 |
| `summary` | 목록/추천 상품 | 상품명, 썸네일, 짧은 가격 정보 |
| `full` | 상세 본문 전체가 필요할 때 | 긴 소개글, 안내문, 이미지, 추천 상품 |

프런트 기본값은 `reservation`입니다. 상세페이지 첫 렌더링에서 긴 본문 DB 필드까지 한 번에 요구하지 않는 쪽이 좋습니다.

## 예약 가능일 API

`GET /api/products/:id/availability`는 기간을 제한합니다.

- 권장 기본 범위: 현재 월 기준 2~3개월
- 프런트 안전 제한: 최대 120일
- 더 먼 날짜는 사용자가 다음 월/다음 일정으로 이동할 때 추가 조회

응답은 날짜별 최소 상태만 내려줍니다.

```json
{
  "id": "2026-07-29",
  "date": "2026-07-29",
  "weekday": "수",
  "status": "available",
  "displayLabel": "예약 가능",
  "remainingSeats": 12
}
```

상세 설명, 긴 안내문, 관리자 메모는 availability 응답에 포함하지 않습니다.

## DB 쿼리 기준

상품 상세:

- `g5_write_product`에서 필요한 컬럼만 선택
- 데일리투어 옵션은 `tour_fee`에서 현재 상품 `wr_id` 기준으로 조회
- 세미패키지 일정은 `v2_pkgTour`에서 노출/삭제 상태를 먼저 필터링

예약 가능일:

- 데일리투어는 `tour_closed_2`, `tour_reg_count` 중심
- 세미패키지는 `v2_pkgTour` 일정 중심
- 긴 기간 전체를 한 번에 조회하지 않음

마이페이지:

- 목록에서는 요약 정보만 조회
- 상세 클릭 시 필요한 결제/여권/룸정보를 추가 조회

## 캐싱 기준

짧게 캐싱해도 되는 데이터:

- 상품명, 썸네일, 기본 가격
- 여권/룸정보 필요 여부
- 판매 중인 옵션 목록

캐싱을 조심해야 하는 데이터:

- 잔여석
- 마감 상태
- 예약 상태
- 결제 상태

## 백엔드 코드 가독성 기준

- API 파일은 endpoint별로 짧게 유지합니다.
- SQL 조립, 응답 변환, 권한 검증을 함수로 분리합니다.
- 기존 PHP helper를 재사용하되, 화면 HTML 출력 함수와 API JSON 응답 함수는 섞지 않습니다.
- 에러 코드는 `backend-api-contract.md`의 공통 코드를 사용합니다.

## 하지 않을 것

- 상세페이지 첫 진입에서 모든 상품 본문과 전체 예약 가능일을 한 번에 내려주지 않음
- 프런트에서 받은 금액을 DB 저장 기준으로 신뢰하지 않음
- 관리자 화면 CSS와 사용자 API 구현을 같은 작업으로 묶지 않음
- 기존 DB 테이블을 대규모로 즉시 마이그레이션하지 않음
